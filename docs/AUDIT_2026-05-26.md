## 🔴 ТОП-7 критичных проблем (бизнес-эффект очевиден)

### 1. Расчёт долга арендатора — 3 разных результата в зависимости от страницы

Один и тот же арендатор может видеть **разные суммы долга** в 5 разных местах из-за непоследовательного фильтра по `deletedAt`.

| Место | Фильтр deletedAt | Файл:строка | Результат |
|---|---|---|---|
| `/admin/dashboard/owner` (топ-должников) | ✓ `chargeScope(orgId)` включает | `app/admin/dashboard/owner/page.tsx:144` | Корректно |
| `/cabinet/documents/print/requisites` (квитанция) | ✓ явно `deletedAt: null` | `app/cabinet/documents/print/requisites/page.tsx:34` | Корректно |
| **`/cabinet/finances`** (что видит сам арендатор) | ❌ нет фильтра | `app/cabinet/finances/page.tsx:35` | **Видит больший долг** |
| **`/admin/tenants/[id]`** (что видит админ) | ❌ нет фильтра | `app/admin/tenants/[id]/page.tsx:231` | **Другая сумма** |
| **`/api/mobile/tenant/finances`** | ❌ нет фильтра | `app/api/mobile/tenant/finances/route.ts:31` | **Третий результат в мобилке** |

**Эффект:** удалил админ ошибочное начисление → арендатор всё равно видит его в долге. Спор с клиентом гарантирован.

### 2. Аддоны не увеличивают лимиты — клиент платит, фича не работает

`lib/org.ts:347-372` `checkLimit()` читает **только** `plan.maxBuildings/maxTenants/maxUsers`, **игнорируя** `OrganizationAddon.quantity`.

```typescript
const max = org.plan[`max${capitalize(type)}` as keyof typeof org.plan] as number | null
```

**Сценарий:** клиент на Pro (maxBuildings=7) купил аддон `BUILDING_PRO × 5` (5 × 1500₸/мес). Супер-админ активировал. Клиент пытается создать 8-е здание → **«Лимит 7 зданий превышен»**. Платит за фичу, которая физически не работает.

### 3. Пени по двум разным формулам — рассинхрон ручной и автоматической

| Где | Файл:строка | Учёт `PENALTY_GRACE_DAYS=1` |
|---|---|---|
| Ручная кнопка | `app/actions/penalties.ts:31-39` | ❌ нет, начисляет с первого дня |
| Cron автоматический | `app/api/cron/check-deadlines/route.ts:248-254` | ✓ есть, начисляет со второго |

**Эффект:** ручной запуск + cron в один день = двойные пени или их рассинхронизация. Через неделю у арендатора в /cabinet одна сумма, в /admin — другая.

### 4. Suspended-организация **не может** оплатить подписку — бизнес-блокер

`lib/org.ts:156-158` `requireOrgAccess()` редиректит suspended на `/login`. Но **`/admin/subscription`** (страница оплаты) тоже использует этот guard.

**Замкнутый круг:**
> подписка истекла → клиент хочет продлить → жмёт «Оплатить» → перебрасывает на `/login` → не может попасть на оплату → звонит саппорту → саппорт вручную меняет `isSuspended=false`

### 5. CashAccount.balance не пересчитывается при удалении Payment

`app/actions/finance.ts:474-488` soft-delete Payment, но связанный `CashTransaction.paymentId` остаётся, и `CashAccount.balance` НЕ уменьшается.

**Эффект:** удалил админ ошибочный платёж → деньги в кассе остались «висеть». Касса показывает на N тысяч больше, чем реально лежит.

---

### 6. Bank-import не в транзакции

`app/actions/bank-import.ts:145,152` обновляет `charge.isPaid=true` через `db.charge.update` **без транзакции**. Параллельный запуск import + ручной payment → один и тот же charge может быть учтён дважды.

---

### 7. `/admin/tenants/[id]` без `force-dynamic` — устаревшие данные

Все остальные критичные страницы имеют `export const dynamic = "force-dynamic"`. Эта — нет. После `updateTenant` → переход на другую вкладку → **видим старые данные** из кэша до 60 секунд.

## 🟠 Расхождения генераторов документов

Один и тот же арендатор/арендодатель в разных документах получает разные представления.

| Документ | landlord-loader | bankAccounts в include | Сумма аренды | shortName | Сохраняет в `GeneratedDocument`? |
|---|---|---|---|---|---|
| **Договор** | `getOrganizationRequisites()` (async DB-запрос) | ✓ грузит | `calculateTenantMonthlyRent` | `shortName()` локальный | ✓ |
| **Счёт** | `organizationToRequisites()` (sync) | ✓ грузит | `calculateTenantMonthlyRent` | — | ✓ |
| **АВР** | `organizationToRequisites()` | **❌ НЕ грузит** → `.find()` упадёт | `calculateTenantMonthlyRent` или `charge.amount` (разное!) | `shortenFio()` новый | ✓ |
| **Акт сверки** | `organizationToRequisites()` | **❌ НЕ грузит** | `charge.amount` прямо из БД (без fallback) | — | ✓ |
| **Приём-передача** | `getOrganizationRequisites()` | **❌ НЕ грузит** | не использует | `shortenFio()` | **❌ НЕ сохраняет** |

### Ключевые проблемы документов

- **Акт сверки** показывает сумму **из charges**, остальные — **через `calculateTenantMonthlyRent()`**. Если сумма в БД отличается от расчёта (например админ правил `customRate`) — данные расходятся.
- **Handover не архивируется** — через год не найдёшь когда передал помещение.
- **3 генератора упадут** если у арендатора нет primary банк-счёта в новой таблице (используют `bankAccounts` без `include`).
- **`Organization.directorPosition`** хранится в БД, редактируется в `/admin/settings`, но во **всех генераторах** подставлено хардкод-фраза `«директора»` (contracts:206, acts:206).
- **`shortName()` локальный** в contracts vs **`shortenFio()`** из `lib/declension` в АВР/handover — разные алгоритмы.

## 🔵 Безопасность multi-tenant

### Найдена 1 КРИТИЧНАЯ дыра

`lib/org.ts:361-365` в `checkLimit("tenants")`:

```typescript
db.tenant.count({
  where: floorIds.length > 0
    ? { OR: [{ space: { floorId: { in: floorIds } } }, { spaceId: null }] }
    : undefined,   // ← считает ВСЕХ tenants по всей платформе если у орга 0 зданий!
})
```

**Эффект:** новая организация без зданий → лимит «10 арендаторов» применяется к общему числу арендаторов **всех клиентов платформы**. Если на платформе уже 50 арендаторов — новая орг получит «лимит превышен» при первой попытке.

Остальные 27 scope-функций (см. `lib/tenant-scope.ts`) **корректны**.

## 🟣 Уведомления

| Проблема | Эффект |
|---|---|
| **10 типов нельзя замьютить** (`SUBSCRIPTION_*`, `ADDON_*`, `SERVICE_*`, `SERVICE_FEE_INDEXED`) | Пользователь получает их безусловно, в настройках их нет |
| **Дедуп есть только для 2 типов** (`CONTRACT_EXPIRING`, `PAYMENT_DUE`) | Все остальные могут засыпать (повторное создание заявки → 2 уведомления) |
| **SMS** включается при `daysLeft ≤ 3` для подписки и `daysLeft ≤ 7` для контракта | По 8-15 ₸/SMS × 100 арендаторов ежедневно = неконтролируемые расходы |
| `notifyInApp` флаг не применяется | In-app сохраняется всегда, флаг бесполезен |
| Локально создаются типы `MESSAGE` и `MESSAGE_RECEIVED` параллельно | UX-путаница в настройках |
