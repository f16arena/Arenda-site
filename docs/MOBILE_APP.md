# Мобильное приложение Commrent — где что и как устроено

Документ описывает структуру мобильного приложения (`mobile/`), как оно связано с
сайтом, и как с ним работать/собирать. Обновляйте при изменениях архитектуры.

## TL;DR
- **Стек:** Expo (SDK 54) + React Native 0.81 + **expo-router** (файловая маршрутизация), TypeScript strict.
- **Точка входа:** `expo-router/entry` → `mobile/app/_layout.tsx`. (Файла `App.tsx` НЕТ — удалён как мёртвый плейсхолдер.)
- **Бэкенд — общий с сайтом.** Приложение ходит в `/api/mobile/*` той же Next.js-инсталляции, которая пишет/читает **ту же базу Postgres** через `@/lib/db`. Любое изменение (с сайта или из приложения) сохраняется сразу и видно обеим сторонам при следующей загрузке/обновлении.

## Структура каталога `mobile/`
```
mobile/
  app/                      # expo-router (файловые маршруты)
    _layout.tsx             # корневой layout (шрифты, провайдеры, Sentry)
    index.tsx               # ГЛАВНЫЙ экран-оболочка: авторизация, табы, загрузка/кэш данных
    screens/
      login.tsx             # вход (логин/пароль + TOTP), биометрия
      tenant.tsx            # все экраны арендатора
      admin.tsx             # экраны admin/accountant/owner
      more.tsx              # настройки, безопасность, push, Telegram, 2FA
      notifications.tsx     # центр уведомлений
  lib/
    api.ts                  # HTTP-клиент + ВСЕ вызовы /api/mobile/* + токены (SecureStore)
    cache.ts                # локальный кэш (readCache/writeCache/clearMobileCache)
    preferences.ts          # локальные настройки (quiet hours для push)
    sentry.ts               # инициализация Sentry
  types/mobile.ts           # типы данных (зеркало ответов /api/mobile/*)
  app.json                  # конфиг Expo (имя, иконки, expo-router, expo-notifications)
  package.json              # зависимости + скрипты сборки EAS
  tsconfig.json             # alias "@/..." → mobile/*
```

## Роли и экраны
- **Арендатор (TENANT):** дашборд/долг, платежи (+QR-оплата), заявки (+фото), счётчики (подача показаний), документы (просмотр + подпись), чат с админом, уведомления, объявления здания.
- **Админ/Бухгалтер (ADMIN/ACCOUNTANT):** «Сегодня», арендаторы (поиск/детали), заявки (статусы), платёжные отчёты (подтвердить/спор/отклонить), здания, задачи, чат, счётчики, документы (подпись).
- **Владелец (OWNER):** всё выше + сводка owner-overview.
- Маршрутизация по ролям — в `app/index.tsx` (функции рендера табов по `role`).

## Связь с сайтом (синхронизация)
- **Один источник правды — Postgres.** Маршруты `app/api/mobile/**/route.ts` (это код САЙТА, Next.js) используют `@/lib/db` и те же модели Prisma, что и веб-админка/кабинет. Отдельной мобильной БД нет.
- **Сохранение сразу:** мутации из приложения (заявка, показание, отчёт о платеже, подпись, задача и т.д.) пишут в БД немедленно через `/api/mobile/*`.
- **Отображение сразу:** после успешной мутации экран вызывает проп `onChanged()` → в `index.tsx` это `onRefresh` → `load()` + `loadTabData(activeTab, { force: true })`, т.е. **принудительный рефетч** свежих данных (кэш игнорируется). Pull-to-refresh — то же самое.
- **Кэш** (`lib/cache.ts`) — только для мгновенного показа последнего состояния офлайн/при старте; при `force` перезатирается свежими данными.
- **Нюанс:** авто-обновление на ВТОРОМ устройстве (или в вебе) без действия пользователя не происходит — нет websockets/пушей данных. Данные подтягиваются при открытии вкладки / pull-to-refresh / повторном входе. Это ожидаемо; при необходимости real-time — отдельная задача (SSE/websocket).

## Авторизация
- `loginMobile()` → `POST /api/mobile/auth/login` (логин/пароль + опц. TOTP). Токены (`accessToken`/`refreshToken` + сроки) хранятся в **SecureStore**.
- Авто-refresh при 401 (`authFetch` → `refreshMobileSession`).
- Биометрия (Face ID/отпечаток/код) — `expo-local-authentication`, быстрый вход после первого логина.
- **Logout:** сначала `unregisterPushDevice()` (отзыв push-токена, нужна авторизация), затем `logoutMobile()` + очистка токенов/кэша.

## Push-уведомления
- `expo-notifications`. Токен Expo получается в `getExpoPushToken()`.
- **Регистрация при входе:** `onLoggedIn` → `registerPushDevice()` (`POST /api/mobile/push-devices`).
- **Отзыв при выходе:** `onLogout` → `unregisterPushDevice()` (`DELETE /api/mobile/push-devices`).
- Тихие часы (quiet hours) — пока локально (`lib/preferences.ts`); серверная синхронизация — в бэклоге.

## Карта эндпоинтов (≈36, все `/api/mobile/*`)
- **auth:** login, register, logout, me, change-password, forgot-password, verify-email, totp/{enroll,verify,disable}
- **tenant:** overview, finances, requests, messages, meters, documents (+ storage/generated by id), contracts, payments/qr
- **admin:** today, tenants (+[tenantId]), requests, payment-reports, buildings, documents (+generated/[id]), contracts, messages, tasks (+[id]), meters
- **owner:** overview
- **общие:** bootstrap, building-notices, notifications, notification-settings, document-signature-requests (+[id], +[id]/sign), push-devices, security/sessions, profile/telegram

## Запуск и сборка
- Локально: `npm --prefix mobile run start` (Expo dev), `--android` / `--ios` / `--web`.
- Базовый URL API: dev → `http://localhost:3000`, prod → `https://commrent.kz` (override `EXPO_PUBLIC_API_BASE_URL`).
- Типчек: `npm --prefix mobile run typecheck` (или из корня `npm run test:mobile:typecheck`).
- **Сборка/выпуск (EAS):** `build:android:preview` / `build:ios:preview` / `build:*:production`, сабмишн `submit:android:internal` / `submit:ios:testflight`. Требуются аккаунты Apple Developer / Google Play и `eas login` (см. `docs/MOBILE_BETA_READINESS.md`). Это делается с машины владельца, не из этого окружения.

## Что осталось по мобильному
- Серверная синхронизация quiet hours (сейчас локально).
- Выпуск в сторы — внешняя инфра (EAS + Apple/Google), не код.
- Опц. real-time обновление (SSE/websocket) вместо рефетча по требованию.
