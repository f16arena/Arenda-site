# Подписание через eGov Mobile (QR / Кросс) — официальный протокол

Источник: NITEC, «Документация к QR и Кросс подписанию в МГОВ v5» + «Инструкция по
подаче заявки на подключение к сервису». Сервис в Smart Bridge: **QR подписание
посредством eGov Mobile (NITEC-S-5096)**.

Суть: телефон (eGov Mobile) сам ходит на ДВА наших публичных HTTPS-эндпоинта.
ШЭП в потоке подписи не участвует — Smart Bridge нужен только для регистрации/
авторизации ИС и допуска к сервису.

## Поток
1. Наш сайт создаёт сессию подписания, рендерит **QR** = `mobileSign:<API №1 URL>`
   (для того же устройства — диплинк `https://m.egov.kz/mobileSign/?link=<API №1>`;
   в URL API №1 ОБЯЗАТЕЛЕН сегмент `mgovSign`, параметры URL-кодируются).
2. Пользователь сканирует QR в eGov Mobile.
3. eGov Mobile → **GET API №1** → получает метаданные + ссылку на API №2.
4. eGov Mobile → **API №2** (GET/POST по auth_type) → получает документ(ы) на подпись.
5. Пользователь подписывает ЭЦП в телефоне → eGov Mobile **PUT** на API №2 тот же
   JSON, но с подписанными данными (CMS base64 вместо `data`).
6. Мы валидируем подпись → отвечаем 200 `{"status":"success"}` (или 403).
7. eGov делает финальный **PUT** (подтверждение доставки) → отвечаем
   `{"status":"success"}` + опц. `{"backUrl":"<вернуть юзера сюда>"}`.

## API №1 (GET) — ответ JSON
```
{
  "description": "Подписание договора",
  "expiry_date": "2026-08-14T15:52:24.138+0500",   // ISO8601
  "organisation": { "nameRu","nameKz","nameEn","bin" },
  "document": {
    "uri": "<API №2 URL>",
    "auth_type": "Token" | "Eds" | "None",
    "auth_token": "<если Token>"
  }
}
```
- **auth_type=Token** (наш выбор): API №2 зовётся с `Authorization: Bearer <auth_token>`
  (одноразовый токен сессии — генерируем сами, проверяем на API №2).
- auth_type=Eds: API №2 = POST с телом `{ "xml": "<подписанный login XML>" }`.
- auth_type=None: API №2 = GET без авторизации.

## API №2 — документы на подпись (ответ GET/POST), и приём подписи (PUT)
```
{
  "signMethod": "CMS_WITH_DATA",   // наш выбор (CMS со штампом времени НУЦ РК)
  "version": 1,
  "documentsToSign": [
    {
      "id": 123,
      "nameRu": "...", "nameKz": "...", "nameEn": "...",
      "meta": [{ "name": "...", "value": "..." }],
      "document": { "file": { "mime": "application/pdf", "data": "<base64 PDF>" } }
    }
  ]
}
```
- Методы: `CMS_WITH_DATA` (CMS + данные, можно извлечь документ — РЕКОМЕНДУЕМ),
  `CMS_SIGN_ONLY` (CMS без данных), `XML` (поле `documentXml`),
  `SIGN_BYTES_ARRAY`, `MIX_SIGN` (per-документ signMethod).
- mime: `application/pdf` → юзеру показывается PDF; `text/plain` → текст.
- При **PUT** JSON «мутирует»: в `document.file.data` (или `documentXml`) приходит
  подписанная CMS-структура base64. Остальные поля — без изменений.
- Ответ на PUT: 200 + `{"status":"success"}` если подпись валидна, иначе 403.
- Ошибки показываем как `{ "message": "локализованный текст" }`.
- Заголовок `Accept-Language: ru|kk|en`.

## Что строим у себя
- `GET  /api/egov-sign/{sessionId}/mgovSign`  — API №1.
- `GET/PUT /api/egov-sign/{sessionId}/doc`     — API №2 (GET отдаёт документ, PUT принимает подпись).
- Таблица сессий: documentId, одноразовый token, status (pending/signed/failed),
  expiry, backUrl.
- Кнопка «Подписать через eGov Mobile (QR)» → создаёт сессию, показывает QR,
  фронт опрашивает статус.
- Подпись — CMS, как от NCALayer → **переиспользуем нашу проверку (NCANode) и
  saveSignature**.
- Эндпоинты — публичные HTTPS. Для требования «статический IP» можно проксировать
  их через наш VPS (78.40.108.250, Caddy) на Next.js.

## Подключение через Smart Bridge (бюрократия — на владельце)
1. sb.egov.kz → Сервисы → найти NITEC-S-5096 → скачать xsd/xml → «Подключиться».
2. Соглашение → «Подключиться без оказания гос.услуги» → форма заявки.
3. Данные для заявки (подготовить):
   - Наименование ИС (+ добавить ИС, если нет в списке).
   - **Логин системы** (≥3 латиница/цифры), **Пароль тест** (≥8), **Пароль прод** (≥8).
   - **Транспортная ЭЦП**: открытая часть GOST-сертификата (.cer/.crt с pki.gov.kz)
     — можно публичную часть ключа руководителя.
   - **Публичный IP (тест/прод)** — наш VPS 78.40.108.250 (или согласовать).
   - Контур взаимодействия (вероятно «Вне ЕТС» → могут потребовать VPN-туннель —
     уточнить у интегратора на тестировании).
   - Основание для подключения (письмо/протокол) — необязательно.
4. Подписать заявку ЭЦП → владелец сервиса рассматривает (2 раб. дня) → НИТ
   организует тестовую среду → тестирование → **акт тестирования** (инициирует
   клиент) → НИТ подключает к прод.

⚠️ Отказ = заявка завершена; повторно — только новой заявкой.
