## 1. 🔴 Безопасность и утечки данных

### 1.1 Публичный `/api/health/db` раскрывает структуру БД — КРИТИЧНО
**Как сейчас:** эндпоинт без авторизации (комментарий в коде: «без авторизации») возвращает версию БД, список таблиц/колонок, статус миграций.
**Проблема:** любой человек в интернете видит схему — карта для целевых атак.
**Предложение:** доступ только platformOwner или по секретному заголовку (как cron). Мониторингу достаточно `{ok: true}` без деталей.

### 1.2 `rejectContractByTenant` обходит soft-delete — ВЫСОКИЙ
**Как сейчас:** `contract-workflow.ts:580` ищет договор `findUnique({ signToken })` без `deletedAt: null` — удалённый арендодателем договор можно отклонить по старой ссылке; статусы перетираются.
**Предложение:** `findFirst({ signToken, deletedAt: null })` — как уже сделано в соседних функциях.

### 1.3 Мобильное API: скоуп по зданиям не везде — ВЫСОКИЙ
**Как сейчас:** `api/mobile/admin/documents/generated/[id]` проверяет организацию, но не проверяет доступ сотрудника к зданию. Бухгалтер здания А может скачать документы арендаторов здания Б.
**Предложение:** после загрузки документа вызывать существующий `assertMobileTenantAccess()` (lib/mobile-admin.ts). Прогнать тот же чек по остальным mobile/admin роутам.

### 1.4 Rate-limit логина — in-memory — СРЕДНИЙ
**Как сейчас:** лимит 8 попыток / 10 мин живёт в памяти инстанса; на Vercel инстансов несколько → перебор распределяется. Ошибки `TOTP_REQUIRED`/`USER_INACTIVE` позволяют выяснять, существует ли аккаунт.
**Предложение:** центральный rate-limit (Upstash Redis из Marketplace), единый ответ «Неверные данные» на все случаи.

### 1.5 Прочее (низкий приоритет)
- Срок ссылки на подпись 90 дней — много для юридически значимой операции; предлагаю 14–30 дней + токен 32 байта.
- `TOTP secret` в БД открытым текстом — приемлемо для MVP, но в бэклог: шифрование на уровне приложения.
- `basicEmailTemplate`: `body` ждёт уже-экранированный HTML — держится на дисциплине; предлагаю брендированный тип или авто-экранирование.
- Email-трекинг: ID — cuid, перебор нереален. Не проблема (понижено после проверки).

## 2. 💰 Финансовая корректность

### 2.1 FIFO-платёж «теряет» остаток — КРИТИЧНО
**Как сейчас:** платёж гасит начисления целиком, «частично — нельзя» (`finance.ts:117`). Платёж 150₸ при начислениях 100₸+110₸ → закрывается только 100₸, остальные 50₸ нигде не учитываются: платёж записан на 150₸, долг уменьшился на 100₸. Понятия «аванс/переплата» нет. Тот же код продублирован в Kaspi-webhook.
**Предложение:** ввести баланс арендатора (переплата = кредит): остаток платежа хранится и автоматически гасит следующее начисление. Это закроет и кейс «арендатор платит вперёд». Альтернатива-минимум: показывать «нераспределённый остаток» админу.

### 2.2 Акт сверки включает удалённые начисления/платежи — КРИТИЧНО
**Как сейчас:** входящее сальдо и строки акта (`reconciliation-builder.ts:78-79, 68-69`) не фильтруют `deletedAt` — удалённое админом ошибочное начисление попадает в официальный документ, расходясь с долгом на экране.
**Предложение:** добавить `deletedAt: null` во все 4 запроса. Часовая работа.

### 2.3 Экспорты (Excel и 1С) не фильтруют удалённое — ВЫСОКИЙ
**Как сейчас:** `/api/export/finances` и `/api/export/1c` выгружают и soft-deleted строки → итоги не сходятся с выпиской.
**Предложение:** `deletedAt: null` в обоих. Подтверждено проверкой.

### 2.4 Пени могут задвоиться по дням — ВЫСОКИЙ
**Как сейчас:** cron дедуплицирует пеню по `period = сегодня`; вчерашняя неоплаченная пеня с другим period не находится → возможны две пени по одному долгу.
**Предложение:** дедуп по (tenantId, PENALTY, исходное начисление) без привязки к дню.

### 2.5 Депозит и «долг» — нужно решение (новое, связано с нашей фичей)
**Как сейчас:** DEPOSIT-начисление попадает в общий «долг» арендатора, и FIFO-платёж гасит его первым (по dueDate он старейший). Из дохода в P&L депозит исключён, из долга — нет.
**Вопрос на обсуждение:** депозит — это долг? Варианты: (а) считать долгом (это реальная дебиторка) — оставить как есть, только подписать в UI «в т.ч. депозит N₸»; (б) исключить из «долга» везде и показывать отдельно. Возврат депозита отрицательной записью может смущать в выгрузках — предлагаю отдельный тип `DEPOSIT_REFUND`.

### 2.6 Мелочи
- Округление: в одних местах `Math.round(x*100)/100`, в других `roundMoney()` с EPSILON — унифицировать на `roundMoney`.
- Kaspi-webhook FIFO не фильтрует `deletedAt`.
- Экспорт фильтрует по `createdAt`, биллинг — по `period` → одно начисление может попасть в разные месяцы разных отчётов. Договориться: финпериоды всегда по `period`.

## 3. 👔 Владелец: UX и недостающее

### 3.1 Заселение арендатора — данные вводятся до 4 раз
**Как сейчас:** диалог создания → карточка (реквизиты заново) → назначение помещения → конструктор договора. Каждый шаг — отдельная страница.
**Предложение:** мастер из 3 шагов: «Контакты и компания → Помещение и условия (ставка, срок, день оплаты, депозит) → Договор (автосборка) + отправка на подпись». Плюс автонапоминание арендатору, если не подписал за N дней.

### 3.2 Месячный цикл — разрозненные кнопки
**Как сейчас:** «Сгенерировать начисления» и «Сгенерировать счета» — отдельные ручные действия; cron есть, но за фича-флагом; АВР — по одному; пакетной печати нет; чек-листа месяца нет.
**Предложение:**
- Карточка «Цикл месяца» на дашборде: Начисления ✓ → Счета ✓ → Отправка → Оплаты (7/12) → Сверка.
- Кнопка «Скачать все счета/АВР за месяц одним ZIP» (по зданию).
- **Авто/ручной режим по тарифу** (ваше пожелание): флаг `autoInvoiceCron` уже есть в тарифе — расширить его на автосоздание документов (сейчас сделано при подписании договора; добавить ежемесячное) и явный переключатель в настройках организации. Ручные конструкторы остаются всегда.

### 3.3 Долги — нет точечного «напомнить»
**Как сейчас:** массовая рассылка шлёт всем (и только на Starter+), на карточке арендатора кнопки «напомнить об оплате» нет, эскалации нет (только чёрный список «всё или ничего»).
**Предложение:** кнопка «Напомнить» на карточке и в списке должников (письмо с суммой, сроком, реквизитами и QR); фильтр рассылки «только должники > N дней»; настраиваемое автонапоминание на 5-й день просрочки.

### 3.4 Чего нет — предлагаю сделать

| **Индексация аренды** (ежегодное повышение по договору) | Поле «индексация %/дата» в условиях аренды + cron повышает ставку и уведомляет; в договоре пункт уже предусмотрен конструктором |

## 4. 🏠 Арендатор: UX и недостающее

### 4.1 Оплата — главная боль
**Как сейчас:** реквизиты + Kaspi QR + ручное «Я оплатил» с чеком, которое админ руками подтверждает. Kaspi-webhook в коде есть, но полноценной интеграции «нажал — оплатил — само подтвердилось» нет.
**Предложение (по нарастающей):** 1) довести Kaspi-webhook до прода (автоподтверждение по externalRef); 2) оплата картой (Click/Эпэй/Freedom Pay); 3) автоплатёж. Это самый монетизируемый кусок продукта.

### 4.2 Уведомления настроены, но не стреляют — ВЫСОКИЙ
**Как сейчас:** каналы и типы (`PAYMENT_DUE`, `PAYMENT_CONFIRMED`, `CONTRACT_EXPIRING`…) есть, настройки в мобилке есть, но триггеры не везде: создано начисление → push/email **не уходит**; платёж подтверждён → не уходит; новый документ → не уходит (только что добавили для подписанных владельцем).
**Предложение:** пройтись по событиям и дозвонить `notifyUser` в: создание charge (cron и ручное), подтверждение PaymentReport, выпуск документа, approaching contractEnd. Инфраструктура готова — работа на день.

### 4.3 ЭЦП в мобилке — «черновик» наружу
**Как сейчас:** арендатор видит кнопки «SMS черновик» / «ЭЦП черновик» с дисклеймером.
**Предложение:** скрыть за фича-флагом до готовности; в вебе ЭЦП уже работает — в мобилке дать ссылку «подписать в браузере».

### 4.4 Чего нет — предлагаю
- Продление договора из кабинета («срок истекает → Запросить продление»).
- Бейдж непрочитанных сообщений на дашборде кабинета (чат есть, но его не видно).
- Веб-кабинет: профиль и настройки уведомлений (есть только в мобилке).
- Подача показаний счётчиков — есть! Но владельцу некуда жать «выставить по показаниям» одним действием — связать с начислениями.

### 4.5 Убрать/спрятать
- SMS-канал в настройках (выключен, «дорого») — спрятать или пометить «скоро».
- «Чат с управляющим» в мобилке виден даже когда админов нет — скрывать.
