## 🛠️ 2. Исправлено в этой сессии (working tree, не закоммичено)

Все правки в `app/actions/finance.ts` и `app/actions/signatures.ts`. Проверено: `eslint` + `tsc --noEmit` = 0.

### 2.1 Касса не откатывалась при удалении платежа (крит #5 был исправлен лишь формально) — CRITICAL/деньги

Корень: `recordPayment` создавал `CashTransaction` **без `paymentId`** (массивная `$transaction([...])` не давала доступа к id платежа). Поэтому `deletePayment` искал проводку по `paymentId` → пусто → баланс не уменьшался. Касса оставалась завышенной после удаления ручного платежа.

- `recordPayment` → интерактивная транзакция, `cashTransaction.paymentId = created.id`.
- `deletePayment` → декрементит баланс, проводку **оставляет** привязанной (для restore).
- `bulkDeletePayments` → раньше кассу **не трогал вообще**; теперь декрементит баланс по каждой проводке.

### 2.2 Undo (restore) был полностью сломан расширением — НОВЫЙ баг (агенты не нашли)

`restorePayment` / `restoreCharge` искали запись через `findFirst` без `deletedAt` → расширение `lib/db.ts` подставляло `deletedAt:null` → **удалённая запись никогда не находилась**. Кнопка «Отменить» не работала в принципе.

- Оба `findFirst` → `deletedAt: { not: null }` (находит удалённое И отключает авто-фильтр).
- `restorePayment` дополнительно возвращает баланс кассы (`increment`).

### 2.3 Переподпись завершённого договора — MEDIUM

`saveSignature` менял статус на SIGNED без проверки текущего. Добавлен guard (как в `contract-workflow.ts`): если `SIGNED`/`REJECTED` → ошибка «Договор уже завершён».

### 2.4 Мёртвая `addPenalty()` — MEDIUM (cleanup)

Экспортируемый server action, создававший PENALTY charge без `PENALTY_GRACE_DAYS` и без дедупа; нигде не вызывался. Заглушён как `calculatePenalties` (кидает явную ошибку).

### 2.5 Kaspi webhook не гасил Charge (бывший открытый пункт B) — HIGH

`app/api/kaspi/webhook/route.ts` создавал `Payment` (с `externalRef`), но не помечал ни один Charge оплаченным → долг арендатора не уменьшался. Эндпоинт сейчас спящий (Kaspi отложен, [[project_payments_strategy]]), но фикс сделан, чтобы при запуске работало корректно. Теперь Payment + погашение charges в одной транзакции: приоритет — конкретный charge из `reference` (`charge:<id>`), затем FIFO по старым неоплаченным (только целиком). Добавлена обработка гонки дублей webhook'а через unique `externalRef` (P2002 → `duplicate`).

### 2.6 Доработки A·D·E (сессия)

- **D — suspended-владелец может войти и оплатить.** `auth.ts` теперь пускает `isSuspended` на логин только для роли `OWNER` (остальные роли при suspend блокируются как раньше); `requireOrgAccess` уводит suspended-владельца на `/admin/subscription` (а не `/login`), остальные роли — на `/login`. Петли нет (admin-layout не вызывает org-guard). Catch-22 закрыт.
- **A — ревок push-девайса при логауте (серверная часть).** `/api/mobile/auth/logout` принимает `pushToken` → `revokedAt/isActive=false`. ⚠️ Требует доработки RN-клиента (`mobile/`): слать Expo-токен в теле logout-запроса. Без этого — no-op (вреда нет).
- **E3 — флаг `notifyInApp` задействован.** `lib/notify.ts` грузит `notifyInApp` заранее и не создаёт in-app уведомление, если юзер его отключил.
- **E1 — анти-дубль сообщений.** `sendMessage` (broadcast и direct) пропускает повторный сабмит идентичного сообщения за 10 сек (контент-аварно, не по типу).

### 2.7 Bank-import идемпотентность (C)

`applyBankImport` теперь пишет детерминированный `externalRef` = `bankimport:sha1(tenantId|amount|date|description)`. Повторная загрузка того же файла → тот же `externalRef` → unique-конфликт → строка пропускается (раньше дублировались платежи). Trade-off: два РАЗНЫХ платежа с полностью идентичными полями схлопнутся; на практике редко (в выписке обычно есть уникальная ссылка/№ документа в description). При появлении нативного bank-txn-id — заменить хэш на него.

> ⚠️ **Нюанс данных:** проводки, созданные **до** фикса 2.1, имеют `paymentId = null` — удаление таких старых платежей по-прежнему не откатит баланс. Нужен разовый backfill (сматчить `CashTransaction` ↔ `Payment` по tenant+amount+date).

## ⚪ 4. Ложные тревоги (зафиксировать, чтобы не возвращаться)

`lib/db.ts` оборачивает Prisma в `$extends({name:"soft-delete"})`: для `Tenant/Charge/Payment/Contract/GeneratedDocument` методы `findMany/findFirst/count/aggregate/groupBy/updateMany/deleteMany` **автоматически** добавляют `deletedAt:null`, если where не упоминает `deletedAt`. Поэтому следующие «CRITICAL» от агентов — **не баги**:

- «Расхождение долга в `app/cabinet/page.tsx:79,88`» — `db.charge.aggregate` авто-фильтруется.
- «Регрессия долга в `app/api/mobile/tenant/overview`» — то же, авто-фильтр.
- «`/admin/ops` без `deletedAt`», «11 mobile-эндпоинтов без `deletedAt`» — то же.

Также корректно **опровергнуты**: дыра `checkLimit` (исправлена), аддоны не растят лимит (исправлены), `PaymentReport.paymentId` orphan (невозможно по порядку операций в транзакции), SMS-расходы (для `_EXPIRING` стоит `sendSms:false`).

Исключения расширения: `findUnique` и единичные `update`/`delete` НЕ перехватываются (поэтому restore чинится через `findFirst` + `deletedAt:{not:null}`).

## 6. Готовый сниппет для A-RN (мобильный клиент)

Серверная часть (`/api/mobile/auth/logout`) уже принимает `pushToken` и ревокает девайс. Осталось, чтобы RN-клиент слал токен при выходе. Там, где `mobile/` вызывает logout (напр. `mobile/lib/api.ts`), добавить `pushToken` в тело запроса:

```ts
// при выходе из аккаунта
const pushToken = await getStoredExpoPushToken() // тот же токен, что отправляли в POST /api/mobile/push-devices
await api.post("/api/mobile/auth/logout", { refreshToken, pushToken })
```

После этого сервер проставит `push_devices.is_active=false, revoked_at`. Без этого изменения серверная часть — безвредный no-op (девайс просто не ревокается). Делается отдельно, т.к. `mobile/*` сейчас в незакоммиченном WIP.
